# Managed install and GitHub update architecture

## Goals

PlotApp now supports a managed desktop installation model for Linux desktops:

- the app is not launched from the unpacked source directory;
- install/update publish a dedicated payload under a managed install home;
- runtime user data and cache are separated from the payload;
- update never runs `git pull` inside the installed copy;
- update rebuilds from a fresh clone and then switches `current`/`previous` safely.

## Architecture layers

1. **Installed payload**
   - generated by `cmake --install` into a staging prefix;
   - published into `install_home/app/current`;
   - previous published payload moves to `install_home/app/previous`.

2. **Runtime user data**
   - default: `~/.local/share/plotapp/`
   - reserved for future user settings, dictionaries, and runtime state.

3. **Runtime cache**
   - default: `~/.cache/plotapp/`
   - preserved across updates and regular uninstall.

4. **System integration**
   - stable desktop manager in `install_home/system/desktop_manager.sh`;
   - launchers in `~/.local/bin/`;
   - GNOME desktop entry in `~/.local/share/applications/plotapp.desktop`;
   - icon in `~/.local/share/icons/hicolor/scalable/apps/plotapp.svg`.

## Why there are two manifest files

### `installation.manifest`

This is the shell-first canonical manifest. It uses a simple `key=value` format so the installer/updater can read it without Python, `jq`, or any extra parser dependency.

### `installation.json`

This is a JSON mirror of the same metadata. It is meant for diagnostics, support, and future integrations.

The shell manager uses `installation.manifest` as the source of truth and writes `installation.json` atomically alongside it.

## Manifest fields

Current manifest keys include:

- `app_name`, `app_id`
- `app_command`, `cli_command`, `update_command`, `uninstall_command`
- `install_home`
- `current_payload_dir`, `previous_payload_dir`
- `system_manager_path`
- `installed_at`, `installed_version`
- `source_type`
- `source_project_root`
- `source_commit`, `source_commit_short`
- `repo_url`, `branch`
- `update_strategy`
- `build_system`, `build_type`, `build_generator`
- `optional_requested`, `optional_installed`
- `gui_requested`, `gui_installed`, `tests_requested`
- `launcher_path`, `cli_launcher_path`, `update_launcher_path`, `uninstall_launcher_path`
- `desktop_file`, `icon_path`
- `runtime_data_dir`, `runtime_cache_dir`
- `payload_has_gui`, `payload_has_cli`

## Update flow

`update_app.sh` performs the following steps:

1. read `installation.manifest`;
2. load `repo_url` and `branch`;
3. if branch is absent, detect remote HEAD and fall back to `main`;
4. query remote commit with `git ls-remote`;
5. compare installed commit vs remote commit;
6. if different:
   - ask for confirmation unless `--yes` is used;
   - clone the branch into a temporary directory with `git clone --depth 1 --branch ...`;
   - configure/build/install into a staging payload;
   - move current payload to `previous`;
   - promote staging to `current`;
   - rewrite both manifest files atomically;
   - refresh wrappers, icon, and desktop entry.

This gives a clean rebuild path without mutating the installed payload in place.

## Safety model

The implementation uses these safeguards:

- staging publish directory before switching `current`;
- `current` в†’ `previous` fallback on publish failure;
- atomic writes for wrappers and manifests via temp file + `mv`;
- desktop-entry rendering uses escaped placeholder substitution plus quoted `Exec=` values, so launcher paths with spaces or `&` stay valid;
- temporary build/staging/backup artifacts are cleaned automatically on early install failure;
- explicit checks for `git` and `cmake`;
- no Python runtime dependency;
- wrapper overwrite protection for non-managed files.

## User commands

### Install from the current source tree

```bash
./install_app.sh --with-gui
```

### Install CLI-only payload

```bash
./install_app.sh --without-gui
```

### Install and immediately bind GitHub updates

```bash
./install_app.sh --repo-url https://github.com/example/plotapp.git --branch main
```

### Check whether a newer GitHub revision exists

```bash
./update_app.sh --check-only
```

### Bind update source after an archive install

```bash
./update_app.sh --set-repo https://github.com/example/plotapp.git --branch main
```

### Update non-interactively

```bash
./update_app.sh --yes
```

### Uninstall but keep user data

```bash
./uninstall_app.sh --yes
```

### Full uninstall with runtime data purge

```bash
./uninstall_app.sh --purge-data --yes
```

## README section example

```text
Managed install:
  ./install_app.sh --with-gui

Check GitHub for updates:
  ./update_app.sh --check-only

Update to the newest revision:
  ./update_app.sh --yes

Remove the managed install but keep runtime data:
  ./uninstall_app.sh --yes

Remove the managed install and runtime data:
  ./uninstall_app.sh --purge-data --yes
```

## Release discipline recommendation

1. Make code changes locally.
2. Run project tests.
3. Update `VERSION.txt`.
4. Commit and push to GitHub.
5. On the workstation, run `update_app.sh --check-only`.
6. If a newer commit exists, run `update_app.sh --yes`.

## Notes about legacy migration

This tree did not contain an older desktop install/update scheme, so a legacy cleanup script was not added.
If older manual launchers already exist in `~/.local/bin/plotapp*`, remove them before the first managed install or migrate them manually.
